# Комплексный анализ и план рефакторинга (LLM-ИИ-кодинг)

Этот документ содержит рекомендации по реструктуризации кодовой базы для повышения эффективности работы ИИ-ассистентов (LLM) и улучшения поддерживаемости проекта.

## Анализ принципов LLM-ориентированного кодинга

**Принципы:**

1. **Атомарность**: Маленькие файлы легче обрабатывать, редактировать и понимать.
2. **Плоская структура**: Избегание глубокой вложенности облегчает навигацию.
3. **Типизация**: Четкие типы данных служат лучшей документацией.
4. **Отсутствие дублирования**: Меньше кода — меньше вероятность ошибок при обновлении.

---

## Высокий приоритет (Critical)

### 1. Модуляризация локализации (i18n)

* **Проблема**: Файлы `ru.ts` и `uk.ts` содержат более 1200 строк. ИИ часто не может "видеть" их целиком, что приводит к ошибкам при вставке или потере контекста.
* **Что сделать**: Разбить переводы на несколько файлов по смыслу.
  * Создать папки `src/data/locales/ru/` и `uk/`.
  * Разделить на файлы: `home.ts`, `auth.ts`, `courses.ts`, `faq.ts`, `common.ts`.
  * Собирать их в основной индексный файл.
* **Результат**: ИИ сможет читать только нужный блок переводов (например, только `home.ts`), экономя токены и повышая точность.

### 2. Очистка неиспользуемого кода

* **Проблема**: Наличие компонентов, которые были заменены новыми версиями.
* **Что сделать**: Проверить `src/components/home` на наличие старых секций, которые больше не используются в `Home.tsx`.

---

## Средний приоритет (Normal)

### 3. Централизация типов (TypeScript)

* **Проблема**: Некоторые интерфейсы и типы определены прямо внутри компонентов.
* **Что сделать**: Перенести все общие типы (Курсы, Пользователь, Транзакции) в `src/types/`.
* **Результат**: ИИ быстрее находит определения типов, не перебирая все файлы компонентов.

### 4. Унификация UI-компонентов

* **Проблема**: Использование сырых тегов `<button>` или `<input>` с классами Tailwind вместо использования базовых компонентов из `src/components/ui`.
* **Что сделать**: Пройтись по компонентам и заменить стандартные элементы на соответствующие компоненты из Shadcn (Button, Input, Card), сохраняя специфические стили через пропсы или `className`.

---

## Низкий приоритет (Low)

### 5. Переименование для ясности

* **Действие**: Переименовать слишком общие названия.
  * `InfoSection` -> `BrandingSection` или `CompanyInfoSection`.
  * `QuizIntro` -> `PreQuizHook`.
* **Результат**: ИИ будет легче находить правильные файлы по названию, меньше путаясь в "инфо-секциях".

---

## Что не нужно менять

* **Структура страниц (`src/pages`)**: Она логична и соответствует роутингу.
* **Интеграция с Supabase**: Настроена корректно и изолированно.
* **Библиотека анимаций (Framer Motion)**: Используется эффективно во всем проекте.
